Which durable workflow platform should we bet on?

Ten contenders. One 40+ dev team. Benched head-to-head on the same lead-lifecycle workflow — durable sleep, human-in-the-loop, parallel fan-out, idempotency, the lot.

Latest verdict — consultancy-delivery lens
Dev-authored engine Inngest Cleanest TS authoring experience, external engine, 4 processes per client instance.
Client-authored engine Kestra Lightest external footprint on the roster — JVM + Postgres, no Redis, no worker fleet.
In-app library DBOS When the workflow belongs inside a Postgres-backed app you already own.
Reliability reference Temporal Gold-standard observability and durability. Cluster weight justifies itself only at scale.

Delivery model for a consultancy

Processes-per-client-instance · standalone engine vs embedded library

External engine — lightest to heaviest

Runs next to the client's app

  • Kestra2 processes
  • Restate3 processes
  • Prefect3 processes
  • Windmill3 processes
  • Inngest4 processes
  • Hatchet4 processes
  • Trigger.dev5+ processes
  • Temporal5–7 processes
Embedded — lives in the client's codebase

Must be inside the app

  • DBOSLibrary + client's Postgres
  • Vercel WorkflowNext.js plugin only
Library-only platforms cannot be offered as a reusable external service. Strong when the workflow belongs inside one owned app; a blocker for "spin up one engine per client" consultancy models.

Per-platform snapshot

Honest one-liners + key numbers

87/100

Temporal

TS · External engine · Apache 2.0 19.8k

Durability reference implementation. Event-sourced history replay, per-activity retry policies, `workflowId` as idempotency key, best observability UI on the roster.

Build time~3.5 min
Exec time~30s
Ops weight5–7 processes
82.5/100

Prefect

⚠ Python-only · External engine · Apache 2.0 22.2k

`@flow` / `@task` decorators read as plain Python. Cleanest HITL primitive after DBOS (`pause_flow_run(wait_for_input=Model)`). Strong self-host story, real production users.

Build time~5 min
Exec time~37s
BlockerNo TS SDK
82/100

Inngest

TS · External engine · Apache 2.0 5.3k

Best TS DX on the roster. Events → functions, native concurrency/throttle/debounce, `waitForEvent` with timeouts. Verified reliability drops to 3 locally — `--persist` loses timers without external Redis.

Build time~4–11 min
Exec time~9–37s
Ops weight4 processes
80/100

Vercel Workflow

TS · Next.js only · Apache 2.0 1.9k

Two directives (`"use workflow"`, `"use step"`) and nothing else. Authoring experience is unmatched. Cost: Next.js-on-Vercel is the happy path, self-host is documented as not-the-happy-path.

Build time~4 min
Exec time~26s
ConstraintLives in Next.js
78/100

DBOS

TS · In-app library · MIT 1.2k

Library-only — no engine to run, just your Node app + Postgres. Workflow-ID-as-idempotency-key, Postgres-journaled replay, cleanest HITL primitive on the roster. Must live inside the client's app.

Build time~4 min
Exec time~32s
DeployInside client app
77/100

Restate

TS · External engine · BSL 1.1 3.8k

Native workflow-ID idempotency via HTTP 409, SQL journal for step observability, virtual-objects programming model. BSL 1.1 license is the biggest hesitation factor.

Build time~3.5 min
Exec time~60s
ConcernBSL license
70/100

Hatchet

TS · External engine · MIT 6.9k

Transparent open-source task-queue engine. Strong observability UI. Fewer high-level primitives (no native idempotency, token rotation friction). Engine + Postgres + RabbitMQ + workers.

Build time~4 min
Exec time~6s
Ops weight4 processes
68/100

Kestra

⚠ YAML DSL · External engine · Apache 2.0 26.8k

Lightest external engine on the roster — JVM + Postgres, no Redis, no separate workers. Best-in-class observability UI. YAML + Pebble templates are the cost, not everyone's TS-friendly.

Build time~18 min
Exec time~40s
Ops weight2 processes
68/100

Windmill

⚠ JSON DSL · External engine · AGPL/EE 16.3k

Strongest structural runtime fit (ForloopFlow parallel, native cron, first-class secrets). Weakest authoring experience — OpenFlow JSON DSL, no local edit-run-test loop.
